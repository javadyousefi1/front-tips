1. What is Performance?

Performance ultimately means user-perceived performance (UPP) — the quality of system outputs in response to user inputs. Users prefer a responsive, smooth app that processes fewer transactions over a choppy, unresponsive app that processes more transactions.

عملکرد (Performance) به معنای «عملکرد درک‌شده توسط کاربر» است. کاربران ترجیح می‌دهند یک اپ روان و پاسخ‌گو داشته باشند، حتی اگر تراکنش‌های کمتری پردازش کند — نه یک اپ ناهموار که تراکنش بیشتری انجام می‌دهد.

1.1 Responsiveness

Responsiveness is how fast the system provides outputs in response to user inputs. For example, when a user taps the screen, they expect pixels to change immediately. The responsiveness metric is the time elapsed between the tap and the pixel change.

Application startup involves three user-perceived events:

  • First paint — sufficient resources loaded to paint an initial frame
  • Interactive — users can tap a button and the app responds
  • Full load — all content has loaded
پاسخ‌گویی یعنی چقدر سریع سیستم به ورودی کاربر واکنش نشان می‌دهد. مثلاً وقتی کاربر صفحه را لمس می‌کند، انتظار دارد پیکسل‌ها فوری تغییر کنند. هنگام راه‌اندازی یک اپ، سه رویداد مهم وجود دارد: اولین رندر، تعامل‌پذیر شدن و بارگذاری کامل.

1.2 Frame Rate

Frame rate is the rate at which the system changes displayed pixels. Users prefer higher frame rates (e.g., 60 fps over 10 fps) for smoother motion perception.

Note Humans usually cannot perceive differences above 60 Hz, which is why most modern displays refresh at that rate.
نرخ فریم تعداد بار در ثانیه‌ای است که سیستم پیکسل‌های نمایش را تغییر می‌دهد. کاربران نرخ‌های بالاتر (مثل ۶۰ fps) را روان‌تر حس می‌کنند. انسان‌ها معمولاً تفاوتی بالاتر از ۶۰ هرتز درک نمی‌کنند، به همین دلیل اکثر نمایشگرهای مدرن با همین نرخ کار می‌کنند.

1.3 Memory Usage

Memory usage doesn't directly impact user perception but closely approximates "user state." A well-designed system uses as much memory as possible to maintain user state while meeting other UPP goals. Free memory is an unused resource.

مصرف حافظه مستقیماً بر کاربر تأثیر نمی‌گذارد ولی «وضعیت کاربر» را حفظ می‌کند. یک سیستم خوب تا حد امکان از حافظه استفاده می‌کند تا وضعیت کاربر حفظ شود. حافظهٔ آزاد یک منبع استفاده‌نشده محسوب می‌شود.

1.4 Power Usage

Users perceive power usage indirectly through device battery life. The system must use only the minimum power required to meet UPP goals.

کاربران مصرف انرژی را از طریق عمر باتری دستگاه احساس می‌کنند. سیستم باید فقط حداقل انرژی لازم برای رسیدن به اهداف عملکرد را مصرف کند.

2. Platform Performance Optimizations

2.1 Web Technologies

HTML and CSS provide high-level declarative languages, increasing productivity with automatic reflow, system theming, and built-in support for different resolutions and languages.

Canvas offers pixel-level control and precise frame-rate control but requires developers to handle multiple resolutions, orientations, and languages.

SVG offers benefits between HTML/CSS and Canvas.

HTML و CSS زبان‌های اعلانی سطح بالا هستند که reflow خودکار، تم سیستم و پشتیبانی از رزولوشن‌های مختلف را فراهم می‌کنند. Canvas کنترل دقیق‌تری روی پیکسل‌ها دارد ولی پیچیدگی بیشتری برای توسعه‌دهنده ایجاد می‌کند. SVG بین این دو قرار می‌گیرد.

2.2 Gecko Rendering Optimizations

JIT Compilation: Gecko's JavaScript engine supports just-in-time compilation, enabling performance comparable to Java virtual machines.

Graphics pipeline optimizations:

  • Reduced invalidation and repainting for common cases (scrolling)
  • Minimized copies when drawing to display framebuffer
  • Special graphics buffer memory directly accessed by compositor hardware
  • GPU rendering for complex scenes
  • Dedicated composition hardware for simple scenes

Animations and Transitions: CSS animations and transitions are offloaded to the system compositor for efficient, power-efficient rendering.

موتور رندر Gecko (در فایرفاکس) از کامپایل JIT برای جاوااسکریپت استفاده می‌کند که سرعت آن را به سطح JVM می‌رساند. همچنین pipeline گرافیکی آن بهینه‌سازی‌هایی مثل کاهش repaint، استفاده از GPU برای صحنه‌های پیچیده و offload کردن انیمیشن‌ها به compositor سیستم را شامل می‌شود.

3. Application Performance

3.1 Startup Performance: The Critical Path

To paint the first frame containing HTML and CSS, the browser must:

  1. Parse HTML
  2. Construct the DOM
  3. Load and decode resources (images)
  4. Apply CSS styles
  5. Reflow the document

The key trick: Run only code on the critical path. Defer work using lazy loading of unnecessary content.

Strategies:

  • Simplify the initial scene
  • Lazily load non-critical resources
  • Front-load requests to avoid waiting for data later
  • Use Service Workers for local caching / offline apps
برای رندر اولین فریم، مرورگر باید HTML را parse کند، DOM بسازد، منابع را بارگذاری کند، CSS اعمال شود و reflow انجام شود. نکته کلیدی: فقط کد مسیر بحرانی (critical path) را اجرا کنید. سایر موارد را با lazy loading به تعویق بیندازید. استفاده از Service Worker برای کش محلی هم توصیه می‌شود.

3.2 Frame Rate

Choose the right tool for content type:

  • HTML/CSS — for mostly static, scrolled, and infrequently animated content
  • Canvas — for highly dynamic content requiring tight rendering control

Firefox optimizes transitions/animations that:

  • Use pages/panels approximately device screen size or smaller
  • Animate only the CSS transform and opacity properties
برای محتوای ایستا از HTML/CSS و برای محتوای بسیار پویا از Canvas استفاده کنید. فایرفاکس انیمیشن‌هایی را که فقط از transform و opacity استفاده می‌کنند، بهتر بهینه می‌کند.

3.3 Memory and Power Usage

Optimization strategies:

  • Don't perform unneeded work
  • Lazily load uncommonly-used UI resources
  • Use efficient data structures
  • Optimize images well
  • Avoid constant timers that prevent CPU low-power modes
  • Listen for the visibilitychange event when apps go to background
برای بهینه‌سازی حافظه و مصرف انرژی: از کارهای غیرضروری خودداری کنید، منابع UI را lazy load کنید، از ساختار داده‌های کارآمد استفاده کنید، تصاویر را فشرده کنید، از تایمرهای دائمی پرهیز کنید و رویداد visibilitychange را برای مدیریت حالت پس‌زمینه گوش دهید.

4. Specific Coding Tips for Application Performance

4.1 Use CSS Animations and Transitions

Instead of library functions using setTimeout() or absolute positioning, use CSS animations and CSS transitions.

Benefits:

  • Browser optimizes and uses GPU automatically
  • Minimal processor impact
  • Standardized syntax
  • Granular control via keyframes
  • Easily triggered with :hover, :focus, :target, or JS class changes
به جای استفاده از setTimeout() یا موقعیت‌دهی absolute در جاوااسکریپت، از CSS animations و CSS transitions استفاده کنید. مرورگر این‌ها را GPU-accelerated می‌کند و پردازنده کمتری مصرف می‌کنند.

4.2 Use CSS Transforms

Instead of tweaking top/left absolute positioning, use the transform CSS property or individual transform properties. 

/* ❌ Avoid: triggers layout recalculation */
element.style.left = '50px';
element.style.top  = '100px';

/* ✅ Prefer: GPU-accelerated, no layout cost */
transform: translate(50px, 100px);

/* Or with individual properties (modern CSS) */
translate: 50px 100px;
rotate: 45deg;
scale: 1.2;
Tip On some platforms you may need translateZ(0.1) to force hardware acceleration, but overuse can cause memory issues.
به جای تغییر top و left که باعث recalculation layout می‌شوند، از transform استفاده کنید. این ویژگی hardware-accelerated است و هیچ هزینهٔ layout ندارد. می‌توانید از properties جداگانه مثل translate، rotate و scale هم استفاده کنید.

4.3 Use requestAnimationFrame() Instead of setInterval()

// ❌ Don't use — wastes CPU, reduces battery life
setInterval(() => {
  // animate something
}, 16);

// ✅ Use requestAnimationFrame instead
function animate() {
  // animate something
  window.requestAnimationFrame(animate);
}
animate();

Why requestAnimationFrame is better:

  • Waits until the browser is ready to render a new frame
  • Won't run while the app is in the background — saves battery
  • Synced to the display refresh rate (no over-rendering)
به جای setInterval از requestAnimationFrame استفاده کنید. این تابع منتظر می‌ماند تا مرورگر آماده رندر شود، در پس‌زمینه اجرا نمی‌شود (در نتیجه باتری ذخیره می‌شود) و با نرخ refresh صفحه همگام است.

4.4 Make Events Immediate

On mobile, click events have a ~300 ms delay. Use touch events for immediate response: 

// ❌ Slow on mobile (300 ms delay)
element.addEventListener('click', handleClick);

// ✅ Immediate on mobile
element.addEventListener('touchstart', handleStart);
element.addEventListener('touchend',   handleEnd);
Tip Always test for touch support first to maintain keyboard/mouse accessibility. Libraries like fastclick handle this gracefully.
در موبایل، رویداد click تقریباً ۳۰۰ میلی‌ثانیه تأخیر دارد. برای پاسخ‌دهی فوری از touchstart و touchend استفاده کنید. ولی حتماً ابتدا پشتیبانی لمسی را چک کنید تا accessibility خراب نشود.

4.5 Keep Your Interface Simple

Problems with complex interfaces:

  • Moving many DOM elements makes everything sluggish
  • Gradients and drop shadows are performance-heavy

Solutions:

  • Simplify look-and-feel
  • During drag-and-drop, move only a proxy element
  • Keep in the DOM only visible elements plus a few on either side
  • Store data in JavaScript objects rather than reading from the DOM
  • Think of the display as a presentation of data, not the data itself

Example — Long list (like tweets): Keep only visible items in the DOM. When scrolling, update the content of the topmost/bottommost elements rather than moving hundreds of invisible nodes. This is far more efficient.

رابط کاربری پیچیده کند است: حرکت دادن عناصر زیاد، gradient‌ها و سایه‌ها هزینه دارند. راه‌حل: DOM را ساده نگه دارید. در لیست‌های بلند (مثل توییتر) فقط آیتم‌های مرئی را در DOM نگه دارید. داده‌ها را در آبجکت‌های JS نگه دارید و DOM را فقط برای نمایش استفاده کنید.

5. General Application Performance Analysis

5.1 Browser Developer Tools

  • Firefox Network Monitor — Shows precise timeline of network requests, request size, load time, and a visual graph of resource loading.
  • Firefox JavaScript Profiler — Identifies the slowest lines of code and shows which functions consume the most time.
  • Built-in Gecko Profiler — More detailed than the JS profiler; shows which browser code runs slowly and provides comprehensive performance data.
Note These tools work with Android Firefox via about:debugging.
فایرفاکس چندین ابزار تحلیل عملکرد دارد: Network Monitor برای بررسی درخواست‌های شبکه، JavaScript Profiler برای یافتن کندترین کدها، و Gecko Profiler که جزئیات بیشتری ارائه می‌دهد. این ابزارها از طریق about:debugging روی موبایل هم کار می‌کنند.

5.2 Mobile-Specific Considerations

Mobile browsers are slower at:

  • Making dozens/hundreds of network requests
  • Rendering large images and CSS gradients
  • Downloading large files (hardware often can't use full bandwidth)
مرورگرهای موبایل در سه زمینه کندتر هستند: درخواست‌های شبکه زیاد، رندر تصاویر بزرگ و gradient های CSS، و دانلود فایل‌های حجیم. سخت‌افزار موبایل اغلب نمی‌تواند از تمام پهنای باند موجود استفاده کند.

5.3 Test Cases and Submitting Bugs

When tools don't reveal the problem:

  1. Create a reduced test case that maximally isolates the issue
  2. Try reproducing with a static HTML copy (including embedded resources)
  3. Remove any private information
  4. Share files or a URL for help
  5. Include profiling information from the tools above
  6. Submit a Bugzilla report or host on a server
وقتی ابزارها مشکل را پیدا نکردند: یک test case کوچک بسازید که مشکل را ایزوله کند، با یک کپی HTML استاتیک آزمایش کنید، اطلاعات خصوصی را حذف کنید و همراه با اطلاعات profiling، گزارش باگ ثبت کنید.
