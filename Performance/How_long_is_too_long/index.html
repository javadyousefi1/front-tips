1. Load Goal — ~1 Second

The very first visual indication that content is loading should appear within 1 second of the user initiating the request. This can be as minimal as displaying the page title and background color — a signal that the browser has received and started processing the response.

Realistic targets vary by network conditions:

  • ~1.5 seconds on a typical office T1 / broadband connection
  • ~5 seconds over a mobile 3G network

Browsers begin rendering HTML as soon as the first bytes arrive — they do not wait for all assets to finish downloading. Leveraging this progressive rendering is key to meeting the 1-second threshold even on slower connections.

اولین نشانه بصری از بارگذاری صفحه باید در کمتر از ۱ ثانیه نمایش داده شود؛ حتی نمایش عنوان صفحه و رنگ پس‌زمینه کافی است. بسته به نوع شبکه، هدف واقعی‌تر عبارت است از: ۱.۵ ثانیه روی خطوط اداری T1 و ۵ ثانیه روی شبکه‌های موبایل 3G. مرورگرها بلافاصله پس از دریافت اولین بایت‌های HTML شروع به رندر می‌کنند — منتظر دانلود کامل همه منابع نمی‌مانند — و همین رفتار «رندر تدریجی» ابزار اصلی دستیابی به این هدف است.
Note The 1-second goal is for the first paint, not for full page load. Full interactivity can take longer, but users need an immediate visual signal to stay engaged.

2. Idle Goal — 50ms Chunks

JavaScript runs on the browser's main thread — the same thread responsible for handling user interactions, painting pixels, and processing layout. Because the main thread is single-threaded, long-running scripts block everything else.

The recommendation is to break JavaScript execution into chunks of 50ms or less during idle periods. Keeping individual tasks under this threshold ensures that:

  • The main thread remains free to respond to user input at any moment.
  • The browser can schedule painting and layout between script tasks.
  • Time-to-interactive (TTI) stays low, since the thread is never blocked for long.

APIs like requestIdleCallback() and chunking via setTimeout() / scheduler.postTask() are designed exactly for this pattern.

مرورگر یک main thread دارد که هم کد JavaScript را اجرا می‌کند، هم رویدادهای کاربر را پردازش می‌کند، و هم رندر صفحه را انجام می‌دهد. از آنجا که این thread تک‌هسته‌ای است، اسکریپت‌های طولانی همه چیز را مسدود می‌کنند. هدف این است که اجرای JavaScript را به تکه‌های حداکثر ۵۰ میلی‌ثانیه تقسیم کنیم تا: thread همیشه برای پاسخ به ورودی کاربر آزاد باشد، مرورگر بتواند بین تسک‌ها رندر کند، و زمان تعامل‌پذیری (TTI) پایین بماند. APIهایی مثل requestIdleCallback() و scheduler.postTask() دقیقاً برای همین الگو طراحی شده‌اند.

2.1 Chunking Example

Instead of processing a large array in one blocking loop, yield control back to the browser between batches using setTimeout with zero delay or scheduler.yield(): 

async function processInChunks(items) {
  const CHUNK_SIZE = 100;
  for (let i = 0; i < items.length; i += CHUNK_SIZE) {
    const chunk = items.slice(i, i + CHUNK_SIZE);
    chunk.forEach(item => processItem(item));

    // yield to the browser — keeps each task ≤ 50ms
    await new Promise(resolve => setTimeout(resolve, 0));
  }
}
به جای پردازش یک آرایه بزرگ در یک حلقه مسدودکننده، کنترل را بین دسته‌ها به مرورگر برمی‌گردانیم. با استفاده از setTimeout(resolve, 0) یا scheduler.yield() در بین هر دسته (chunk)، اجازه می‌دهیم مرورگر رندر کند یا به رویدادهای کاربر پاسخ دهد. این الگو مانع بلوکه‌شدن main thread می‌شود و هر تسک را زیر آستانه ۵۰ میلی‌ثانیه نگه می‌دارد.

3. Animation Goal — 16.7ms (60 fps)

Smooth animations require the browser to produce 60 frames per second (fps). At 60 fps, each frame budget is 1000 ÷ 60 ≈ 16.7 milliseconds.

Within that 16.7ms window, the browser itself needs roughly 6ms to handle compositing and rendering. That leaves approximately 10ms for:

  • JavaScript execution (event handlers, animation callbacks)
  • Style recalculation
  • Layout (reflow)
  • Paint and composite

Whenever the frame rate drops below 60 fps — or frame times are uneven — users perceive the animation as janky. High-refresh-rate displays (90 Hz, 120 Hz) have even tighter budgets: ~11ms and ~8ms respectively.

انیمیشن‌های روان نیاز به ۶۰ فریم در ثانیه (fps) دارند، یعنی بودجه هر فریم حدود ۱۶.۷ میلی‌ثانیه است. از این بودجه، خود مرورگر حدود ۶ میلی‌ثانیه برای compositing مصرف می‌کند، پس تنها ~۱۰ میلی‌ثانیه برای اجرای JavaScript، محاسبه Style، Layout (reflow) و Paint باقی می‌ماند. هر بار که نرخ فریم زیر ۶۰ افت کند یا زمان‌بندی فریم‌ها نامنظم شود، کاربر انیمیشن را «لرزان» (jank) حس می‌کند. نمایشگرهای high-refresh-rate مانند ۹۰ و ۱۲۰ هرتز بودجه تنگ‌تری دارند — حدود ۱۱ و ۸ میلی‌ثانیه.

3.1 Using requestAnimationFrame

Always drive visual updates through requestAnimationFrame() — never with setInterval or setTimeout for animations. The browser calls your callback at the optimal time just before the next paint, so you stay synchronized with the display's refresh cycle and avoid wasted frames. 

function animate(timestamp) {
  // timestamp is DOMHighResTimeStamp — use it for delta calculations
  updateScene(timestamp);
  drawFrame();
  requestAnimationFrame(animate); // schedule next frame
}

requestAnimationFrame(animate); // kick off the loop
برای همه به‌روزرسانی‌های بصری همیشه از requestAnimationFrame() استفاده کنید، نه از setInterval یا setTimeout. مرورگر این callback را دقیقاً قبل از رندر بعدی صدا می‌زند تا انیمیشن با چرخه رفرش نمایشگر هماهنگ بماند و فریم‌های اضافه تولید نشود. پارامتر timestamp از نوع DOMHighResTimeStamp است و برای محاسبه delta بین فریم‌ها استفاده می‌شود.
Tip Prefer CSS animations and transitions for simple effects — the browser can often run them on the compositor thread entirely, bypassing JavaScript overhead and keeping the main thread free.

4. Responsiveness Goal — 50–200ms

Perceived responsiveness is how quickly the UI acknowledges a user's action. The guidelines define two thresholds:

  • Immediate feedback (hover, button press): within 50ms — the user should feel the UI react instantly, with no detectable lag.
  • Complete response: within 100–200ms — the full result of the action (data loaded, navigation complete, etc.) should appear within this window.

If a response will take longer than 100ms, always provide immediate visual feedback — a spinner, loading skeleton, progress bar, or button state change — to acknowledge that the action was received. Without feedback, users assume the UI is frozen and may click again, creating duplicate requests.

پاسخ‌دهی درک‌شده به این بستگی دارد که رابط کاربری چقدر سریع عمل کاربر را تأیید کند. دو آستانه وجود دارد: ۵۰ میلی‌ثانیه برای بازخورد فوری (hover، کلیک دکمه) و ۱۰۰ تا ۲۰۰ میلی‌ثانیه برای پاسخ کامل. اگر پاسخ بیش از ۱۰۰ میلی‌ثانیه طول بکشد، باید بلافاصله یک نشانه بصری نمایش دهید — spinner، skeleton loading، نوار پیشرفت، یا تغییر حالت دکمه — تا به کاربر بگویید درخواستش دریافت شده است. در غیر این صورت کاربر فکر می‌کند UI منجمد شده و دوباره کلیک می‌کند.

4.1 Interaction to Next Paint (INP)

INP (Interaction to Next Paint) is a Core Web Vital that measures the latency of all interactions on a page — clicks, taps, and keyboard input — and reports the worst-case value. Google's thresholds are:

  • Good ≤ 200ms
  • Needs Improvement 201ms – 500ms
  • Poor > 500ms

INP replaced First Input Delay (FID) as a Core Web Vital in March 2024, shifting focus from the first interaction to the overall responsiveness of a page during its entire lifetime.

INP (Interaction to Next Paint) یک Core Web Vital است که تأخیر تمام تعاملات صفحه — کلیک، لمس، و ورودی صفحه‌کلید — را اندازه می‌گیرد و بدترین مقدار را گزارش می‌دهد. آستانه‌های Google: ≤ ۲۰۰ میلی‌ثانیه خوب، ۲۰۱ تا ۵۰۰ میلی‌ثانیه نیاز به بهبود، و > ۵۰۰ میلی‌ثانیه ضعیف است. INP در مارس ۲۰۲۴ جایگزین FID (First Input Delay) شد و تمرکز را از اولین تعامل به پاسخ‌دهی کلی صفحه در طول عمرش تغییر داد.

4.2 Reducing Input Latency

The most effective strategies for staying within the 50–200ms responsiveness window:

  • Avoid long tasks on the main thread. Any task over 50ms delays event processing. Break long tasks with scheduler.yield() or setTimeout(0).
  • Debounce / throttle expensive handlers. Scroll and input events fire at very high frequency; running heavy logic on every event causes jank.
  • Minimize layout thrashing. Interleaving DOM reads and writes forces synchronous reflows. Batch all reads first, then all writes.
  • Use passive event listeners. Marking touch/wheel listeners as { passive: true } lets the browser scroll without waiting for your handler to complete.
  • Offload work to Web Workers. CPU-heavy computation (parsing, filtering, encoding) should run off the main thread so it cannot block interactions.
مؤثرترین راهکارها برای ماندن در پنجره ۵۰ تا ۲۰۰ میلی‌ثانیه عبارتند از: ۱) پرهیز از long task در main thread — هر تسک بیشتر از ۵۰ میلی‌ثانیه پردازش رویداد را به تأخیر می‌اندازد؛ با scheduler.yield() یا setTimeout(0) تسک‌ها را بشکنید. ۲) Debounce/Throttle کردن handlerهای سنگین — رویدادهای scroll و input با فرکانس بالا فیر می‌شوند. ۳) کاهش layout thrashing — خواندن و نوشتن DOM را با هم قاطی نکنید؛ ابتدا همه read، سپس همه write. ۴) استفاده از passive event listener — با { passive: true } اسکرول را بدون انتظار برای handler بلافاصله اجرا کنید. ۵) انتقال کار به Web Worker — محاسبات سنگین (parse، filter، encode) باید خارج از main thread اجرا شوند.
Warning Never assume "my users are on fast devices." INP is measured in the field across all your real users — budget for the slowest common hardware and network, not your dev machine.
