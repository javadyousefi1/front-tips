1. What is Latency?

Latency is the time it takes for a packet of data to travel from source to a destination. In terms of performance optimization, it's important to optimize to reduce causes of latency and to test site performance emulating high latency to optimize for users with lousy connections.

Latency is generally considered to be the amount of time it takes from when a request is made by the user to the time it takes for the response to get back to that user. On a first request, for the first 14KB bytes, latency is longer because it includes a DNS lookup, a TCP handshake, and the secure TLS negotiation. Subsequent requests will have less latency because the connection to the server is already established.

Latency describes the amount of delay on a network or Internet connection. Low latency implies that there are no or almost no delays. High latency implies that there are many delays. One of the main aims of improving performance is to reduce latency.

The latency associated with a single asset, especially a basic HTML page, may seem trivial. But websites generally involve multiple requests: the HTML includes requests for multiple CSS files, scripts, and media files. The greater the number and size of these requests, the greater the impact of high latency on user experience.

On a connection with low latency, requested resources will appear almost immediately. On a connection with high latency, there will be a discernible delay between the time that a request is sent and the resources are returned.

Latency can be measured one way — for example, the time it takes to send a request for resources — or as the length of the entire round-trip from the browser's request for a resource to the moment when the requested resource arrives at the browser.

Latency (تأخیر) زمانی است که یک بسته داده از مبدأ به مقصد می‌رسد. به‌طور کلی‌تر، زمانی است از لحظه درخواست کاربر تا دریافت پاسخ. در اولین درخواست برای ۱۴ کیلوبایت اول، تأخیر بیشتر است چون شامل DNS Lookup، TCP Handshake و TLS Negotiation می‌شود. درخواست‌های بعدی تأخیر کمتری دارند چون اتصال از قبل برقرار شده. Low latency یعنی تقریباً هیچ تأخیری وجود ندارد و High latency یعنی تأخیرهای زیاد. یکی از اهداف اصلی بهبود عملکرد، کاهش latency است. تأخیر یک فایل ساده کم است، اما هر صفحه شامل درخواست‌های متعدد برای CSS، اسکریپت و رسانه است — هر چه تعداد و حجم این درخواست‌ها بیشتر باشد، تأثیر high latency روی تجربه کاربری بیشتر است. latency را می‌توان یک‌طرفه یا به‌صورت round-trip (رفت و برگشت کامل) اندازه گرفت.

2. Network Throttling

To emulate the latency of a low bandwidth network, you can use developer tools and switch to a lower-end network connection. In the DevTools network tab, you can switch the throttling option to 2G, 3G, etc. Different browser developer tools have different preset options. The characteristics emulated include:

  • Download speed
  • Upload speed
  • Minimum latency — the minimum time it takes to send a packet of data
Selection Download speed Upload speed Min latency (ms)
GPRS 50 kbps 20 kbps 500
Regular 2G 250 kbps 50 kbps 300
Good 2G 450 kbps 150 kbps 150
Regular 3G 750 kbps 250 kbps 100
Good 3G 1.5 Mbps 750 kbps 40
Regular 4G/LTE 4 Mbps 3 Mbps 20
DSL 2 Mbps 1 Mbps 5
Wi-Fi 30 Mbps 15 Mbps 2
Tip Always test your site under throttled network conditions — especially GPRS (500ms latency) and Regular 3G (100ms) — to experience what users on slow connections face.
برای شبیه‌سازی تأخیر شبکه‌های کم‌پهنا، از DevTools استفاده کنید و Throttling را روی 2G، 3G و غیره تنظیم کنید. هر preset سه ویژگی را شبیه‌سازی می‌کند: سرعت دانلود، سرعت آپلود و حداقل تأخیر (ms). مقادیر کلیدی جدول: GPRS: دانلود ۵۰ kbps، آپلود ۲۰ kbps، تأخیر ۵۰۰ms (بسیار بد). DSL: ۲/۱ Mbps، تأخیر ۵ms. Wi-Fi: ۳۰/۱۵ Mbps، تأخیر ۲ms. همیشه سایت را در شرایط شبکه throttled شده آزمایش کنید تا تجربه کاربران با اتصال کند را درک کنید.

3. Network Timings

In the DevTools network tab, you can see how long each request took to complete — broken down into distinct phases. Each phase represents a specific source of latency that you can inspect and optimize independently.

در تب Network در DevTools می‌توانید ببینید هر درخواست چقدر طول کشیده — به‌تفکیک مراحل مختلف. هر مرحله یک منبع جداگانه از تأخیر است که می‌توانید مستقلاً آن را بررسی و بهینه کنید.

3.1 Blocked

When a request is in a queue, waiting for a network connection, it is considered blocked. Blocking happens when there are too many simultaneous connections made to a single server over HTTP. If all connections are in use, the browser can't download more resources until a connection is released — meaning those requests and resources are stuck waiting.

Blocked یعنی درخواست در صف انتظار است و به اتصال شبکه‌ای دسترسی ندارد. این زمانی رخ می‌دهد که تعداد اتصالات همزمان به یک سرور از طریق HTTP به حداکثر رسیده باشد. تا زمانی که یک اتصال آزاد نشود، مرورگر نمی‌تواند منابع بیشتری دانلود کند.

3.2 DNS Resolution

DNS resolution is the time it took to do the DNS lookup. The greater the number of unique hostnames on a page, the more DNS lookups need to be done, and the more time is spent in this phase.

DNS Resolution مدت زمانی است که برای انجام DNS Lookup صرف شده. هر چه تعداد hostname‌های منحصربه‌فرد در صفحه بیشتر باشد، DNS Lookup‌های بیشتری لازم است و زمان بیشتری در این مرحله صرف می‌شود.

3.3 Connecting

Connecting is the time it takes for a TCP handshake to complete. Like DNS, the greater the number of server connections needed, the more time is spent creating those connections.

Connecting زمان لازم برای تکمیل TCP Handshake است. مثل DNS، هر چه تعداد اتصالات به سرورهای مختلف بیشتر باشد، زمان بیشتری برای برقراری این اتصال‌ها صرف می‌شود.

3.4 TLS Handshake

TLS handshake is how long it took to set up a secure connection. While a TLS handshake does take longer to connect than an insecure connection, the additional time needed for a secure connection is worth it.

TLS Handshake زمان راه‌اندازی اتصال امن HTTPS است. این مرحله از یک اتصال ناامن کندتر است، اما ارزشش را دارد چون داده‌ها رمزنگاری شده و از شنود جلوگیری می‌کند.

3.5 Sending

Sending is the time taken to send the HTTP request to the server. This is typically very short — it measures only the time to transmit the request itself, not the response.

Sending زمانی است که برای ارسال HTTP request به سرور صرف می‌شود. معمولاً بسیار کوتاه است — فقط زمان انتقال خود درخواست را اندازه می‌گیرد، نه زمان پاسخ.

3.6 Waiting (Disk Latency)

Waiting is disk latency — the time it took for the server to complete its response. This phase used to be the main area of performance concern. However, server performance has improved significantly as computer memory and CPU have improved. Depending on the complexity of what is needed from the server (e.g., database queries, server-side rendering), this can still be a significant issue.

Waiting یا Disk Latency زمانی است که سرور برای آماده کردن پاسخ صرف می‌کند. در گذشته این مرحله بزرگ‌ترین مشکل عملکردی بود، اما با بهبود سخت‌افزار سرورها (CPU و RAM) بهتر شده. با این حال، بسته به پیچیدگی عملیات سمت سرور (مثل query پایگاه داده یا رندر سمت سرور)، هنوز هم می‌تواند مشکل‌ساز باشد.

3.7 Receiving

Receiving is the time it takes to download the asset. The receiving time is determined by a combination of network capacity and asset file size.

  • If the image had been cached, this phase would have been nearly instantaneous
  • On a throttled network connection, receiving a single image could take up to 43 seconds
Insight The receiving phase is directly within your control as a developer — optimizing asset size (compression, modern formats like WebP/AVIF, lazy loading) directly reduces this time.
Receiving زمان دانلود فایل است که توسط دو عامل تعیین می‌شود: ظرفیت شبکه و حجم فایل. اگر فایل cache شده باشد، این مرحله تقریباً فوری است. در شبکه throttled شده، دانلود یک تصویر می‌تواند تا ۴۳ ثانیه طول بکشد! این مرحله مستقیماً در کنترل توسعه‌دهنده است — بهینه‌سازی حجم فایل‌ها (فشرده‌سازی، فرمت‌های مدرن مثل WebP/AVIF، lazy loading) مستقیماً این زمان را کاهش می‌دهد.

4. Measuring Latency

There are two distinct types of latency worth measuring:

  • Network latency — the time it takes for a data request to get from the computer making the request to the computer responding, including the time for a byte of data to make the return trip back. It is generally measured as a round-trip delay.
  • Disk latency — the time it takes from the moment a computer (usually a server) receives a request to the time it returns the response. This is purely the server-side processing time, independent of network transmission.
دو نوع latency وجود دارد که باید اندازه گرفته شوند: Network latency — زمانی که یک درخواست داده از کامپیوتر درخواست‌کننده به کامپیوتر پاسخ‌دهنده می‌رسد و بایت‌های داده برمی‌گردند. این معمولاً به‌صورت Round-Trip Delay اندازه گرفته می‌شود. Disk latency — زمانی که از لحظه دریافت درخواست توسط سرور تا ارسال پاسخ طول می‌کشد. این کاملاً زمان پردازش سمت سرور است و مستقل از انتقال شبکه‌ای است.
