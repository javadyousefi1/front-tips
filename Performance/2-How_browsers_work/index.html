1. Overview

Fast sites provide better user experiences. Users want and expect web experiences with content that is fast to load and smooth to interact with.

Two major issues in web performance are:

  1. Issues having to do with latency
  2. Issues having to do with the fact that browsers are single-threaded

Latency is the biggest threat to ensuring a fast-loading page. It is the developers' goal to make the site load as fast as possible — or at least appear to load super fast — so the user gets the requested information as quickly as possible. Network latency is the time it takes to transmit bytes over the air to computers.

For the most part, browsers are single-threaded. They execute a task from beginning to end before taking up another task. For smooth interactions, the developer's goal is to ensure performant site interactions — from smooth scrolling to being responsive to touch. Render time is key, ensuring the main thread can complete all the work and still always be available to handle user interactions. Web performance can be improved by understanding the single-threaded nature of the browser and minimizing the main thread's responsibilities to ensure rendering is smooth and responses to interactions are immediate.

سایت‌های سریع تجربه بهتری ارائه می‌دهند. دو مشکل اصلی در عملکرد وب وجود دارد: اول تأخیر (Latency) که بزرگ‌ترین تهدید برای سرعت بارگذاری است و به زمان انتقال بایت‌ها از طریق شبکه گفته می‌شود. هدف توسعه‌دهنده اینست که سایت واقعاً یا حداقل به‌نظر سریع باشد. دوم تک‌نخی بودن مرورگر (Single-threaded): مرورگر یک وظیفه را از ابتدا تا انتها اجرا می‌کند و تا اتمام آن به وظیفه دیگری نمی‌پردازد. بنابراین thread اصلی باید همیشه آزاد باشد تا به تعاملات کاربر پاسخ دهد. بهبود عملکرد یعنی به حداقل رساندن مسئولیت‌های thread اصلی.

2. Navigation

Navigation is the first step in loading a web page. It occurs whenever a user requests a page by:

  • Entering a URL into the address bar
  • Clicking a link
  • Submitting a form
  • Other actions

One of the goals of web performance is to minimize the amount of time navigation takes to complete. In ideal conditions, this usually doesn't take too long, but latency and bandwidth are foes that can cause delays.

Navigation اولین مرحله بارگذاری یک صفحه وب است و هر بار که کاربر آدرسی وارد می‌کند، روی لینکی کلیک می‌کند، فرمی ارسال می‌کند یا اقدام مشابهی انجام می‌دهد رخ می‌دهد. یکی از اهداف عملکرد وب، به حداقل رساندن زمان navigation است. در شرایط ایده‌آل سریع است، اما تأخیر شبکه و پهنای باند می‌توانند آن را کند کنند.

2.1 DNS Lookup

The first step of navigating to a web page is finding where the assets for that page are located. If you navigate to https://example.com, the HTML page is located on the server with IP address 93.184.216.34. If you've never visited this site, a DNS lookup must happen.

Your browser requests a DNS lookup, which is eventually fielded by a name server, which in turn responds with an IP address. After this initial request, the IP will likely be cached for a time, which speeds up subsequent requests by retrieving the IP address from the cache instead of contacting a name server again.

DNS lookups usually only need to be done once per hostname for a page load. However, DNS lookups must be done for each unique hostname the requested page references. If your fonts, images, scripts, ads, and metrics all have different hostnames, a DNS lookup will have to be made for each one.

This can be problematic for performance, particularly on mobile networks. When a user is on a mobile network, each DNS lookup has to go from the phone to the cell tower to reach an authoritative DNS server. The distance between a phone, a cell tower, and the name server can add significant latency.

اولین قدم، پیدا کردن آدرس IP سرور است. مرورگر یک DNS Lookup درخواست می‌کند و name server آدرس IP را برمی‌گرداند. این IP معمولاً برای مدتی cache می‌شود تا درخواست‌های بعدی سریع‌تر باشند. نکته مهم: DNS lookup برای هر hostname منحصربه‌فرد در صفحه باید انجام شود — فونت‌ها، تصاویر، اسکریپت‌ها، تبلیغات و metrics اگر hostname متفاوت داشته باشند، هر کدام به یک DNS lookup جداگانه نیاز دارند. در شبکه‌های موبایل این مشکل‌سازتر است چون سیگنال باید از گوشی به آنتن و از آنتن به DNS server برود که تأخیر قابل توجهی ایجاد می‌کند.

2.2 TCP Handshake

Once the IP address is known, the browser sets up a connection to the server via a TCP three-way handshake. This mechanism is designed so that two entities attempting to communicate — in this case the browser and web server — can negotiate the parameters of the network TCP socket connection before transmitting data, often over HTTPS.

TCP's three-way handshaking technique is often referred to as "SYN-SYN-ACK" — or more accurately SYN, SYN-ACK, ACK — because there are three messages transmitted by TCP to negotiate and start a TCP session between two computers. This means three more messages back and forth between each server, and the request has yet to be made.

پس از یافتن IP، مرورگر از طریق TCP Three-Way Handshake اتصال برقرار می‌کند. این فرآیند شامل سه پیام است: SYN → SYN-ACK → ACK. هدف آن توافق بر روی پارامترهای اتصال قبل از ارسال داده است. یعنی قبل از اینکه هنوز هیچ درخواستی ارسال شده باشد، سه پیام رفت‌وبرگشت بین مرورگر و سرور انجام می‌شود.

2.3 TLS Negotiation

For secure connections established over HTTPS, another "handshake" is required — the TLS negotiation. It determines which cipher will be used to encrypt the communication, verifies the server, and establishes that a secure connection is in place before beginning the actual transfer of data. This requires five more round trips to the server before the request for content is actually sent.

While making the connection secure adds time to the page load, a secure connection is worth the latency expense, as the data transmitted between the browser and the web server cannot be decrypted by a third party.

Note After the eight round trips to the server (3 TCP + 5 TLS), the browser is finally able to make the actual content request.
برای اتصالات HTTPS، یک TLS Negotiation اضافی لازم است. این مذاکره تعیین می‌کند از چه cipher رمزنگاری استفاده شود، سرور را تأیید می‌کند و اتصال امن برقرار می‌کند. این فرآیند پنج رفت‌وبرگشت اضافه به سرور نیاز دارد. در کل، قبل از ارسال اولین درخواست واقعی، ۸ رفت‌وبرگشت (۳ مرحله TCP + ۵ مرحله TLS) انجام می‌شود. با اینکه امنیت زمان می‌برد، ارزشش را دارد چون داده‌ها توسط اشخاص ثالث رمزگشایی نمی‌شوند.

3. Response

Once we have an established connection to a web server, the browser sends an initial HTTP GET request on behalf of the user, which for websites is most often an HTML file. Once the server receives the request, it will reply with relevant response headers and the contents of the HTML. 

<!doctype html>
<html lang="en-US">
  <head>
    <meta charset="UTF-8" />
    <title>My simple page</title>
    <link rel="stylesheet" href="styles.css" />
    <script src="myscript.js"></script>
  </head>
  <body>
    <h1 class="heading">My Page</h1>
    <p>A paragraph with a <a href="https://example.com/about">link</a></p>
    <div>
      <img src="my-image.jpg" alt="image description" />
    </div>
    <script src="another-script.js"></script>
  </body>
</html>

This response contains the first byte of data received. Time to First Byte (TTFB) is the time between when the user made the request — say by clicking on a link — and the receipt of this first packet of HTML. The first chunk of content is usually 14KB of data.

The linked resources aren't requested until the browser encounters the links during parsing, even if the total page is larger than the initial 14KB packet.

پس از برقراری اتصال، مرورگر یک HTTP GET request ارسال می‌کند و سرور با HTML پاسخ می‌دهد. TTFB (Time to First Byte) زمانی است که از کلیک کاربر تا دریافت اولین بایت HTML طول می‌کشد. اولین بسته داده معمولاً ۱۴ کیلوبایت است. منابع لینک‌شده (CSS، JS، تصاویر) تا زمانی که مرورگر در حین parse کردن به آن‌ها برسد درخواست داده نمی‌شوند.

3.1 Congestion Control / TCP Slow Start

TCP packets are split into segments during transmission. Because TCP guarantees the sequence of packets, the server must receive an acknowledgment (ACK) from the client after sending a certain number of segments.

In order to balance the number of transmitted segments, the TCP slow start algorithm is used to gradually increase the amount of transmitted data until the maximum network bandwidth can be determined, and to reduce the amount of transmitted data in case of high network load.

The number of segments to be transmitted is controlled by the value of the congestion window (CWND), which can be initialized to 1, 2, 4, or 10 MSS (MSS is 1500 bytes over the Ethernet protocol).

  • If an ACK is received → CWND value is doubled, so the server can send more segments next time
  • If no ACK is received → CWND value is halved to reduce load

This mechanism achieves a balance between sending too many segments and sending too few, adapting to the actual network capacity dynamically.

TCP Slow Start الگوریتمی است که حجم داده ارسالی را به‌تدریج افزایش می‌دهد تا حداکثر پهنای باند شبکه تعیین شود. تعداد بخش‌های قابل ارسال توسط congestion window (CWND) کنترل می‌شود که می‌تواند با ۱، ۲، ۴ یا ۱۰ MSS (هر MSS برابر ۱۵۰۰ بایت) شروع شود. اگر ACK دریافت شود، CWND دو برابر می‌شود و سرور می‌تواند بیشتر بفرستد. اگر ACK دریافت نشود، CWND نصف می‌شود. این مکانیسم به‌صورت پویا با ظرفیت واقعی شبکه تطبیق پیدا می‌کند و از ارسال بیش از حد یا کمتر از حد جلوگیری می‌کند.

4. Parsing

Once the browser receives the first chunk of data, it can begin parsing the information received. Parsing is the step the browser takes to turn the data it receives over the network into the DOM and CSSOM, which is used by the renderer to paint a page to the screen.

Even if the requested page's HTML is larger than the initial 14KB packet, the browser will begin parsing and attempting to render an experience based on the data it has. This is why it's important for web performance optimization to include everything the browser needs to start rendering a page — or at least a template of the page — the CSS and HTML needed for the first render — in the first 14KB.

به محض دریافت اولین بسته داده، مرورگر شروع به Parsing می‌کند. Parsing فرآیند تبدیل داده‌های دریافتی از شبکه به DOM و CSSOM است که renderer برای رسم صفحه از آن‌ها استفاده می‌کند. حتی اگر صفحه بزرگ‌تر از ۱۴ کیلوبایت اولیه باشد، مرورگر فوراً شروع به parse و render می‌کند. به همین دلیل بسیار مهم است که همه چیزی که برای رندر اولیه لازم است — CSS و HTML — در همان ۱۴ کیلوبایت اول قرار بگیرد.

4.1 Building the DOM Tree

The first step is processing the HTML markup and building the DOM tree. HTML parsing involves tokenization and tree construction. HTML tokens include start and end tags, as well as attribute names and values. If the document is well-formed, parsing it is straightforward and faster.

The DOM tree describes the content of the document. The <html> element is the first element and root node of the document tree. The tree reflects the relationships and hierarchies between different elements. Elements nested within other elements are child nodes. The greater the number of DOM nodes, the longer it takes to construct the DOM tree.

When the parser finds non-blocking resources, such as an image, the browser will request those resources and continue parsing. Parsing can continue when a CSS file is encountered, but <script> elements — particularly those without an async or defer attribute — block rendering and pause the parsing of HTML. Though the browser's preload scanner hastens this process, excessive scripts can still be a significant bottleneck.

اولین قدم، پردازش HTML و ساخت DOM tree است. HTML parsing شامل دو مرحله است: Tokenization (تقسیم به توکن‌های start tag، end tag، attribute و مقدار) و Tree Construction (ساختن درخت سلسله‌مراتبی). عنصر <html> root node درخت است. هر چه تعداد DOM node‌ها بیشتر باشد، ساختن DOM tree طولانی‌تر است. منابع non-blocking مثل تصاویر بارگذاری می‌شوند و parsing ادامه می‌یابد، اما تگ <script> بدون async یا defer هم رندر و هم parsing HTML را متوقف می‌کند.

4.2 Preload Scanner

While the browser builds the DOM tree, this process occupies the main thread. As this happens, the preload scanner will parse through the content available and request high-priority resources like CSS, JavaScript, and web fonts. Thanks to the preload scanner, we don't have to wait until the parser finds a reference to an external resource to request it. It will retrieve resources in the background so that by the time the main HTML parser reaches the requested assets, they may already be in flight or have been downloaded. 

<link rel="stylesheet" href="styles.css" />
<script src="my-script.js" async></script>
<img src="my-image.jpg" alt="image description" />
<script src="another-script.js" async></script>

In this example, while the main thread is parsing the HTML and CSS, the preload scanner will find the scripts and image, and start downloading them. To ensure the script doesn't block the process, add the async attribute, or the defer attribute if JavaScript parsing and execution order is important.

Note Waiting to obtain CSS doesn't block HTML parsing or downloading, but it does block JavaScript because JavaScript is often used to query CSS properties' impact on elements.
در حالی که thread اصلی مشغول ساخت DOM است، Preload Scanner به‌طور موازی محتوای HTML را اسکن می‌کند و منابع با اولویت بالا مثل CSS، JavaScript و فونت‌ها را در پس‌زمینه دانلود می‌کند. این باعث می‌شود وقتی parser اصلی به این منابع می‌رسد، احتمالاً قبلاً دانلود شده باشند. برای جلوگیری از blocking اسکریپت‌ها، از attribute async (برای اسکریپت‌های مستقل) یا defer (برای اسکریپت‌هایی که ترتیب اجرا مهم است) استفاده کنید. نکته مهم: CSS دانلود HTML را block نمی‌کند، اما JavaScript را block می‌کند چون JS اغلب به CSS properties دسترسی دارد.

4.3 Building the CSSOM Tree

The second step in the critical rendering path is processing CSS and building the CSSOM tree. The CSS object model is similar to the DOM — both are trees and independent data structures. The browser converts the CSS rules into a map of styles it can understand and work with. The browser goes through each rule set in the CSS, creating a tree of nodes with parent, child, and sibling relationships based on the CSS selectors.

The CSSOM tree includes styles from the user agent style sheet. The browser begins with the most general rule applicable to a node and recursively refines the computed styles by applying more specific rules — in other words, it cascades the property values.

Building the CSSOM is very fast, and its build time is not displayed separately in developer tools. Rather, the "Recalculate Style" in DevTools shows the total time to parse CSS, construct the CSSOM tree, and recursively calculate computed styles. In terms of web performance, the total time to create the CSSOM is generally less than the time it takes for one DNS lookup.

دومین قدم در مسیر رندر، ساخت CSSOM tree است. مثل DOM، CSSOM هم یک درخت مستقل است. مرورگر قوانین CSS را به یک نقشه از استایل‌ها تبدیل می‌کند و برای هر selector یک گره در درخت می‌سازد با روابط parent، child و sibling. CSSOM شامل استایل‌های user agent (مرورگر پیش‌فرض) هم هست. مرورگر از کلی‌ترین قانون شروع می‌کند و با قوانین خاص‌تر آن را override می‌کند — یعنی همان cascade. ساخت CSSOM بسیار سریع است و در DevTools زیر "Recalculate Style" نشان داده می‌شود. معمولاً سریع‌تر از یک DNS Lookup انجام می‌شود.

4.4 JavaScript Compilation

While the CSS is being parsed and the CSSOM created, other assets including JavaScript files are downloading (thanks to the preload scanner). JavaScript is parsed, compiled, and interpreted. The scripts are parsed into abstract syntax trees (AST). Some browser engines take the ASTs and pass them into a compiler, outputting bytecode. This is known as JavaScript compilation. Most of the code is interpreted on the main thread, but there are exceptions such as code run in web workers.

در حین parse و ساخت CSSOM، فایل‌های JavaScript هم (توسط preload scanner) دانلود می‌شوند. JS سه مرحله را طی می‌کند: Parsing (تبدیل به Abstract Syntax Tree یا AST)، Compilation (برخی موتورها AST را به bytecode کامپایل می‌کنند) و Interpretation (اجرا). اکثر کدها روی thread اصلی اجرا می‌شوند، به استثنای کدهایی که در Web Worker اجرا می‌شوند.

4.5 Building the Accessibility Tree (AOM)

The browser also builds an accessibility tree that assistive devices use to parse and interpret content. The accessibility object model (AOM) is like a semantic version of the DOM. The browser updates the accessibility tree when the DOM is updated. The accessibility tree is not modifiable by assistive technologies themselves.

Important Until the AOM is built, the content is not accessible to screen readers.
مرورگر همچنین یک Accessibility Tree می‌سازد که ابزارهای کمکی مثل screen reader‌ها از آن برای parse و تفسیر محتوا استفاده می‌کنند. AOM (Accessibility Object Model) نسخه معنایی (semantic) DOM است. هرگاه DOM بروز شود، AOM هم بروز می‌شود. ابزارهای کمکی نمی‌توانند مستقیماً AOM را تغییر دهند. نکته حیاتی: تا زمانی که AOM ساخته نشده، محتوا برای screen reader‌ها قابل دسترسی نیست.

5. Render

Rendering steps include style, layout, paint, and in some cases compositing. The CSSOM and DOM trees created in the parsing step are combined into a render tree which is then used to compute the layout of every visible element, which is then painted to the screen. In some cases, content can be promoted to its own layer and composited, improving performance by painting portions of the screen on the GPU instead of the CPU, freeing up the main thread.

مرحله Render شامل چهار زیرمرحله است: Style → Layout → Paint → Compositing. DOM و CSSOM با هم ترکیب شده و یک Render Tree می‌سازند که برای محاسبه layout هر عنصر مرئی استفاده می‌شود، سپس روی صفحه نقاشی می‌شود. در برخی موارد محتوا به لایه جداگانه‌ای ارتقا می‌یابد که روی GPU رندر می‌شود — این کار عملکرد را بهبود می‌دهد و thread اصلی را آزاد می‌کند.

5.1 Style

The third step in the critical rendering path is combining the DOM and CSSOM into a render tree. Construction starts with the root of the DOM tree, traversing each visible node.

Elements that aren't going to be displayed — like the <head> element and any nodes with display: none — are not included in the render tree as they will not appear in the rendered output. However, nodes with visibility: hidden applied are included in the render tree, as they do take up space.

Each visible node has its CSSOM rules applied to it. The render tree holds all the visible nodes with content and computed styles — matching up all the relevant styles to every visible node in the DOM tree, and determining, based on the CSS cascade, what the computed styles are for each node.

سومین قدم: ترکیب DOM و CSSOM برای ساخت Render Tree. مرورگر از root شروع می‌کند و هر node مرئی را traverse می‌کند. عناصری مثل <head> یا هر چیزی با display: none در render tree نیستند. اما عناصر با visibility: hidden هستند چون فضا اشغال می‌کنند. در نهایت هر node مرئی با computed style خودش (بر اساس cascade CSS) در render tree قرار می‌گیرد.

5.2 Layout

The fourth step in the critical rendering path is running layout on the render tree to compute the geometry of each node. Layout is the process by which the dimensions and location of all the nodes in the render tree are determined, plus the determination of the size and position of each object on the page. Reflow is any subsequent size and position determination of any part of the page or the entire document.

On the web page, almost everything is a box. Different devices and different desktop preferences mean an unlimited number of differing viewport sizes. In this phase, taking the viewport size into consideration, the browser determines what the sizes of all the different boxes are going to be on the screen. Layout generally starts with the body, laying out the sizes of all the body's descendants.

The first time the size and position of each node is determined is called layout. Subsequent recalculations are called reflows. For example, if the dimensions of an image are not declared, a reflow will occur once the image dimensions are known after loading.

چهارمین قدم: Layout — محاسبه هندسه (ابعاد و مکان) هر node در render tree. هر چیزی در وب تقریباً یک جعبه (box) است. مرورگر با در نظر گرفتن اندازه viewport تعیین می‌کند هر box چه ابعاد و چه موقعیتی روی صفحه داشته باشد. Layout معمولاً از body شروع می‌شود و به زیرعناصر ادامه می‌یابد. اولین بار که مکان و اندازه node‌ها تعیین می‌شود Layout نام دارد و محاسبات بعدی Reflow. مثلاً اگر ابعاد تصویر تعریف نشده باشد، بعد از بارگذاری تصویر یک Reflow رخ می‌دهد.

5.3 Paint

The last step in the critical rendering path is painting the individual nodes to the screen, the first occurrence of which is called the First Meaningful Paint. In the painting or rasterization phase, the browser converts each box calculated in the layout phase to actual pixels on the screen. Painting involves drawing every visual part of an element to the screen, including text, colors, borders, shadows, and replaced elements like buttons and images.

To ensure smooth scrolling and animation, everything occupying the main thread — including calculating styles, reflow, and paint — must take the browser less than 16.67ms to accomplish (at 60 fps). At 2048 × 1536, the iPad has over 3,145,000 pixels to be painted to the screen — a lot of pixels that have to be painted very quickly.

Painting can break the elements in the layout tree into layers. Promoting content into layers on the GPU (instead of the main thread on the CPU) improves paint and repaint performance. Elements that instantiate their own layer include:

  • <video>
  • <canvas>
  • Any element with CSS property opacity
  • Any element with 3D transform
  • Any element with will-change
Warning Layers improve performance but are expensive in memory management. Do not overuse them as a web performance optimization strategy.
آخرین قدم: Paint (رنگ‌آمیزی) — تبدیل هر جعبه محاسبه‌شده به پیکسل‌های واقعی روی صفحه. اولین بار این رخ دهد First Meaningful Paint نامیده می‌شود. Paint شامل رسم متن، رنگ، border، shadow و عناصر جایگزین مثل دکمه و تصویر است. برای اسکرول و انیمیشن روان، همه چیز در thread اصلی باید زیر ۱۶.۶۷ میلی‌ثانیه انجام شود (۶۰ fps). برای مثال iPad 2048×1536 بیش از ۳,۱۴۵,۰۰۰ پیکسل دارد که باید بسیار سریع رسم شوند. برای بهینه‌سازی، مرورگر می‌تواند عناصر را به لایه‌های جداگانه تبدیل کند و آن‌ها را روی GPU رندر کند. عناصری مثل <video>، <canvas>، و عناصر با opacity، 3D transform یا will-change لایه خودشان را دارند. هشدار: لایه‌ها حافظه بیشتری مصرف می‌کنند، بنابراین نباید بیش از حد استفاده شوند.

5.4 Compositing

When sections of the document are drawn in different layers, overlapping each other, compositing is necessary to ensure they are drawn to the screen in the right order and the content is rendered correctly.

As the page continues to load assets, reflows can happen. A reflow sparks a repaint and a re-composite. Had we defined the dimensions of our image, no reflow would have been necessary, and only the layer that needed to be repainted would be repainted, and composited if necessary. But when image dimensions are not declared and the image is obtained from the server, the rendering process goes back to the layout steps and restarts from there.

وقتی بخش‌های مختلف صفحه در لایه‌های جداگانه رسم می‌شوند، Compositing تضمین می‌کند که لایه‌ها به ترتیب صحیح روی هم قرار گیرند. اگر در حین بارگذاری assets یک Reflow رخ دهد، یک repaint و re-composite هم لازم می‌شود. راه‌حل جلوگیری از این مشکل: همیشه ابعاد تصاویر را در HTML تعریف کنید تا وقتی تصویر از سرور می‌رسد، نیازی به Reflow نباشد و فقط لایه مربوطه repaint شود.

6. Interactivity

Once the main thread is done painting the page, you would think we would be "all set." That isn't necessarily the case. If the load includes JavaScript that was correctly deferred and only executed after the onload event fires, the main thread might be busy, and not available for scrolling, touch, and other interactions.

Time to Interactive (TTI) is the measurement of how long it took from the first request — which led to the DNS lookup and TCP connection — to when the page is interactive. Interactive means the point in time after the First Contentful Paint when the page responds to user interactions within 50ms. If the main thread is occupied parsing, compiling, and executing JavaScript, it is not available and therefore not able to respond to user interactions in a timely fashion.

For example: maybe the image loaded quickly, but a another-script.js file was 2MB and the user's network connection was slow. In this case, the user would see the page super quickly, but wouldn't be able to scroll without jank until the script was downloaded, parsed, and executed. Avoid occupying the main thread with heavy JavaScript.

حتی بعد از اینکه paint تمام شد، ممکن است صفحه هنوز تعامل‌پذیر نباشد. اگر JavaScript deferred بعد از onload اجرا شود، thread اصلی مشغول است و نمی‌تواند به scroll، لمس یا کلیک پاسخ دهد. TTI (Time to Interactive) زمانی است که از اولین درخواست (DNS Lookup) تا لحظه‌ای که صفحه می‌تواند به تعاملات کاربر در کمتر از ۵۰ میلی‌ثانیه پاسخ دهد اندازه می‌گیرد. این نقطه بعد از First Contentful Paint است. مثال: اگر another-script.js فایل ۲ مگابایتی باشد و اتصال کاربر کند، صفحه خیلی زود نمایش داده می‌شود اما کاربر نمی‌تواند اسکرول کند یا با آن تعامل داشته باشد تا وقتی که اسکریپت دانلود، parse و اجرا نشود — تجربه کاربری بدی است. از بار سنگین گذاشتن روی thread اصلی با JavaScript حجیم پرهیز کنید.
